## 1. Verdict en une phrase

> **VCBench est un prototype de recherche solide et simple à déployer, sur lequel un vernis « production » (authentification, migrations Alembic, file de jobs/observabilité) a été ajouté récemment — mais ce vernis n'est pas branché de bout en bout.** La base (monolithe FastAPI + Dash + pipeline bioinfo conteneurisé) est cohérente ; les couches d'entreprise existent en façade et sont court-circuitées, à moitié câblées, ou contredites par la documentation.

## 2. Portrait du système

### 2.1 En une ligne
Plateforme de contrôle qualité de séquençage génome entier (WGS) qui ingère des runs DRAGEN (upload ZIP navigateur ou import AWS S3), benchmarke leurs VCF contre les *truth sets* GIAB avec **hap.py** (petits variants) et **Truvari** (variants structuraux) en conteneurs, persiste les métriques dans **PostgreSQL**, et affiche résultats + progression des jobs dans un dashboard co-hébergé.

### 2.2 Architecture
Monolithe **délibéré** : un seul process uvicorn (`start_app.sh` → `uvicorn api.app.main:app`, défaut `127.0.0.1:8002`) sert l'API REST **et** l'UI.

Dans `api/app/main.py` : l'app FastAPI ajoute CORS puis `SecurityHeadersMiddleware`, inclut neuf routers sous `/api/v1` (le router de données Dash sous `/api/v1/dash`), déclare deux endpoints WebSocket (`/ws/download/{sample_id}`, `/ws/jobs/{job_id}`), et **monte enfin l'app Dash (WSGI) à `/`** via `WSGIMiddleware` (`main.py:82`). Starlette résolvant les routes dans l'ordre d'enregistrement, les `/api/v1/*` et `/docs` explicites passent avant le catch-all Dash.

Trois couches derrière l'API :
1. **Persistance** : `models.py` (ORM SQLAlchemy), `crud.py` (toutes lectures/écritures, commit/rollback par appel — chaque writer commit inline plutôt que via la Session de requête de `get_db`), `schemas.py` (Pydantic, `from_attributes=True`).
2. **Orchestration** : `job_service.py` enveloppe `crud` pour gérer le cycle de vie `TransferJob` (create/progress/phase/complete/fail/cancel/retry) et ajoute un `TransferEvent` à chaque transition.
3. **Travail lourd** : `api/tasks/process_run.py` valide références + fichiers de run, filtre/normalise les VCF avec bcftools/tabix, lance les wrappers Docker `pipeline/happy.sh` et `pipeline/truvari.sh`, puis parse les sorties (`api/tasks/parsers.py`) et écrit les métriques.

Le **frontend Dash** (`dash_app/`) est une SPA : `dcc.Location` route `/`, `/runs`, `/home`, `/monitoring`, `/truvari` vers des builders de layout, et les callbacks `@callback` récupèrent les données en HTTP depuis l'API co-hébergée (`API_BASE_URL`, défaut `http://127.0.0.1:8002/api/v1`). **Certaines lectures contournent la DB et lisent directement le répertoire de sorties sur disque.**

**Auth** : clés d'API par header (`X-VCBench-API-Key`) mappées à un palier de rôle (operator < reviewer < admin) dans `security.py` ; les routes mutantes dépendent de `require_role(...)`. **Crucial** : `get_current_role` retourne ADMIN dès que `VCBENCH_AUTH_DISABLED` est vrai OU qu'aucune clé n'est configurée (`security.py:49`) — donc la posture par défaut, sortie de la boîte, est **totalement ouverte**.

### 2.4 Cycle de vie d'une requête
Deux points d'entrée d'ingestion convergent vers le même pipeline, plus un déclencheur manuel de benchmarking.

- **Upload ZIP navigateur** (`POST /api/v1/upload/runs`) : `require_role(OPERATOR)` (no-op si auth off). Nom de fichier nettoyé, `TransferJob` (type UPLOAD_ZIP, phase UPLOAD) créé, fichier streamé sur disque en chunks de 8 Mo (1 `TransferEvent` PROGRESS par chunk, `MAX_UPLOAD_BYTES` appliqué mid-stream). `LabRun` créé/réinitialisé à PENDING_PROCESSING. Si `auto_process`, `BackgroundTasks` planifie `process_run_background` → EXTRACT → PROCESSING → unzip sûr → PROCESS → `run_pipeline`. Succès → AWAITING_APPROVAL ; exception → rollback + FAILED + message.
- **Import AWS S3** (`POST /api/v1/upload/aws`) : crée un job QUEUED AWS_IMPORT et (si `auto_process`) planifie `process_aws_run_background`, qui streame `aws_download_gvcf.sh` ligne par ligne (miroir vers le store ws en mémoire **et** un `TransferEvent`), puis `ensure_references` (phase REFERENCE_SETUP) et `run_pipeline` (phase PROCESS).
- **Pipeline** (`run_pipeline`) : hap.py → résout l'échantillon de base, assure les références, localise VCF/BED/FASTA/FAI (crée un SDF RTG si absent), filtre le GVCF aux régions de l'index FASTA (bcftools+tabix), construit les chemins Docker `/wgs`, lance `happy.sh`, puis `parse_summary` retourne **seulement la ligne SNP/ALL** → un `HappyMetric` écrit. Truvari → filtre + normalise chr des VCF SV et BED, lance `truvari.sh`, `parse_truvari_summary` → un `TruvariMetric`. CSV → écrit des `QCMetric`. Toutes les écritures de métriques résolvent `run_id` via `crud.get_lab_run_by_name`.
- **Résultats** : `/monitoring` poll `/jobs/summary`, `/jobs`, `/jobs/{id}`, `/jobs/{id}/events` toutes les 3 s. `/runs` et `/truvari` lisent `/runs` + `/benchmarking` + endpoints TruvariMetric (certains panneaux QC lisent le disque directement). Un reviewer approuve via `POST /runs/{id}/approve`.

> ⚠️ **Note** : le benchmarking manuel (`POST /runs/{run_name}/benchmarking`) et `POST /runs/upload` exécutent `run_pipeline` **inline dans la requête**, bloquant le worker pour toute la durée du benchmark ; seuls `/upload/runs` et `/upload/aws` utilisent `BackgroundTasks`.

### 2.5 Modèle de données
Toutes les tables dans une seule base PostgreSQL (défaut host port 55433) :

- **`users`** : id, username (unique), full_name, hashed_password (PBKDF2-SHA256). Référencé par `lab_runs.approved_by_id`.
- **`lab_runs`** *(entité centrale)* : id, run_name (unique), status (enum `PENDING_PROCESSING → PROCESSING → AWAITING_APPROVAL → APPROVED`, ou `FAILED`), timestamps, error_message, approved_by_id → users.id. Possède en cascade (`all, delete-orphan`) : `qc_metrics`, `happy_metrics`, `truvari_metrics`.
- **`qc_metrics`** : id, metric_name, metric_value, file_source, run_id. *En pratique, écrit avec name/value NULL et seulement file_source → essentiellement un pointeur de fichier.*
- **`happy_metrics`** : une ligne par run = la ligne SNP/ALL de hap.py (type, filter, totaux truth/query, TP/FN/FP/UNK, recall, precision, frac_na, F1, ratios TiTv et het/hom).
- **`truvari_metrics`** : une ligne par run (UNIQUE(run_id)) : tp_base/tp_comp/fp/fn, precision/recall/f1, base_cnt/comp_cnt, gt_concordance, breakdowns génotype.
- **`transfer_jobs`** : id (UUID string), type, subject_id, status (queued/running/completed/failed/canceled), phase, source_uri, destination_path, champs bytes/rate/eta, timestamps, error, cancel_requested, metadata JSON. Possède `transfer_events` (cascade).
- **`transfer_events`** : log append-only par job ; séquence monotone (UNIQUE(job_id, sequence)), timestamp, level, phase, message, champs bytes/rate, metadata JSON.

**Relations** : `lab_runs` 1—N {qc,happy,truvari}_metrics (truvari contraint à 1/run) ; `lab_runs` N—1 `users` (approbateur) ; `transfer_jobs` 1—N `transfer_events`. Les jobs de transfert ne sont liés aux runs **que par une chaîne** `subject_id`/`run_name`, **pas par une FK**.

### 2.6 Forces réelles
- Déploiement vraiment simple : un seul process pour l'API + l'UI.
- Persistance en couches propre : `models`/`schemas`/`crud` s'accordent sur noms/types.
- Uploads streamés et bornés (chunks de 8 Mo sur disque, `MAX_UPLOAD_BYTES` mid-stream, cleanup en cas d'échec).
- Hygiène de sécu par endroits : PBKDF2-SHA256 600k itérations + sel par mot de passe, comparaison constante via `secrets.compare_digest`, hiérarchie de rôles, `SecurityHeadersMiddleware` (nosniff/Referrer-Policy/X-Frame-Options).
- Forte observabilité des opérations longues : chaque opération crée un `TransferJob` + log `TransferEvent` séquencé, exposé sur `/monitoring`.
- Pipeline bioinfo reproductible et épinglé à des conteneurs, avec normalisation VCF délibérée (filtrage régions, renommage chr, indexation tabix).
- **Le README racine est exact** et vérifié contre le code (port 8002, DB 55433, env `vcbench`, route `/monitoring` présente).

## 3. Risques majeurs (top 8)

1. **Auth ouverte par défaut** : `get_current_role` retourne ADMIN si `VCBENCH_AUTH_DISABLED` OU aucune clé (`security.py:49-50`). Un déploiement qui oublie `VCBENCH_API_KEYS` est totalement ouvert — upload, import AWS, delete, approve tournent en admin.
2. **Le front ne peut pas s'authentifier contre sa propre API** : les callbacks POST côté serveur (`runs.py:492` benchmark, `:608` import AWS) appellent `requests.post` **sans** header de clé. Dès que de vraies clés sont configurées, les boutons de l'UI renvoient 401/403 — l'UI ne marche que tant que l'API est non protégée.
3. **Deux chemins de création de schéma → deux bases différentes** : README utilise `init_db.py` (`create_all`, honore `ondelete='CASCADE'`) ; le chemin Alembic crée les mêmes FK **sans** ondelete (`20260501_0001_initial_schema.py:101,130`). Supprimer une `LabRun` se comporte différemment selon le mode de création, et `create_all` ne pose jamais `alembic_version`.
4. **Travail long bloquant sur les threads de requête** : `POST /runs/{name}/benchmarking` et `POST /runs/upload` appellent `run_pipeline()` inline (`runs.py:60-67, :211-218`).
5. **Plan de contrôle cancel/retry câblé mais inerte** : `jobs.py` expose `/cancel` et `/retry`, `job_service` pose `cancel_requested`, mais **aucun worker ne lit `cancel_requested`** et ces endpoints n'ont **aucun consommateur UI**.
6. **Traitement `BackgroundTasks` non durable** : un job QUEUED ne progresse que si `auto_process` l'a planifié ; un redémarrage perd tout BackgroundTask en vol + le store de logs en mémoire → `LabRun`/`TransferJob` coincés en PROCESSING/RUNNING sans reprise.
7. **Deux canaux de progression parallèles** pour l'import AWS (store ws en mémoire + flux `TransferEvent`), synchronisés à la main → duplication, dérive facile ; et les events PROGRESS par chunk de 8 Mo génèrent un volume d'events/commits énorme sur gros uploads.
8. **Doc dérivée** : les guides `docs/*.md` décrivent une config ancienne (port 8000, DB 5433, env `bioinfo`, chemins `/mnt/acri4_2`) qui contredit le code ; un opérateur les suivant échouera au démarrage.

## 4. Les 5 incohérences structurelles

### 4.1 L'auth existe mais est court-circuitée partout
`security.py` implémente des écritures gatées par rôle, mais (a) le front `dash_app/` n'envoie **aucun** header de clé → activer l'auth casse toute l'UI, (b) la config par défaut (clés vides) renvoie ADMIN à tout le monde, (c) les WebSockets n'ont aucune auth, (d) `hash_password` existe mais **aucun chemin de login** ne l'utilise → toute la machinerie users/mots de passe est **décorative**.

### 4.2 Trois chemins de bootstrap DB qui divergent
`init_db.py` (`create_all`, README) ≠ `create_db_tables.py` (`alembic upgrade head`, ENVIRONMENT_README) ≠ migrations brutes. Symptôme : dérive CASCADE. Les migrations ne survivent à un double-passage que grâce à leurs gardes `_has_table`/`_add_column_if_missing` — couplage fragile.

### 4.3 Le sous-système transfer/jobs est câblé pour un worker async qui n'existe pas
Jobs QUEUED que rien ne démarre ; `cancel_requested` honoré par personne ; retry produit des jobs morts ; `/ws/jobs` keyé par le mauvais id. **Une feature entière livrée sans sa moitié « exécution ».**

### 4.4 Pipeline bloquant sur le thread de requête
Cf. risque #4. Incohérent avec les chemins d'upload qui, eux, utilisent `BackgroundTasks`.

### 4.5 Les lacunes de couverture sont exactement aux endroits les plus risqués
`upload_run.py` (ingestion d'archive non fiable) et `setup_reference.py` (sélection du truth-set — un mauvais mapping benchmarke contre le mauvais GIAB) **n'ont jamais été ouverts en profondeur**. Zéro test sur tout le cœur bioinfo (seuls 3 fichiers de tests transfer existent).

## 5. Findings issus de l'exploration dynamique (en plus de l'audit statique)

Trois interactions manuelles ont touché des zones que l'audit statique a sous-couvertes (`callbacks.py`/`home.py` n'ont été inspectés qu'en surface) :

1. **Pas de bouton d'approbation dans l'UI** *(non listé par l'audit)* — `POST /runs/{id}/approve` existe côté API, mais aucun bouton/callback Dash ne l'appelle. Seul élément lié : une pastille lecture-seule « Approved/Pending » sur `/` (`index.py:149-151`). Une run reste donc bloquée en `AWAITING_APPROVAL` pour qui n'utilise que le dashboard. Complète le finding [API-DATA-1] (« `approved_by_id` jamais rempli »). **Contournement appliqué** : run de démo approuvée via l'API.
2. **Auth ouverte par défaut** — confirmé en live (approbation réussie sans authentification). = risque majeur #1.
3. **🔧 Dashboard QC qui plantait — CORRIGÉ** — `callbacks.py:649`, `disp = f"{ref_val:.4f}"` levait `ValueError: Unknown format code 'f' for object of type 'str'` sur les lignes texte de la catégorie « Summary » (`type=INDEL`, `filter=ALL`). Comme Summary est la catégorie par défaut, le callback `update_table` plantait au chargement (HTTP 500 sur `/_dash-update-component`) → `ref-dropdown.options` jamais peuplé → dashboard vide. **Correctif** : branche `elif isinstance(ref_val, str): disp = ref_val` ajoutée dans les deux boucles de rendu générique (colonnes gauche + droite). Vérifié au navigateur headless (rendu OK, zéro erreur console). **Statut : hotfix local non commité.**

> Balayage de robustesse : les 10 catégories du dashboard ont été cliquées — aucune ne plante (zéro 500, zéro erreur console). Les catégories vides (CNV, ROH, HeThom, Ploidy, SV, BED) le sont parce qu'**aucun fichier source DRAGEN** n'existe pour le run de démo, pas à cause d'un bug.

## 7. Lacunes de couverture (critique de complétude)

Modules **jamais examinés** par l'audit statique, à ouvrir en priorité :

- **`api/tasks/upload_run.py`** (133 l.) — chemin d'ingestion d'archive **non fiable**. Contient bien un durcissement zip-slip/symlink/taille (`safe_extract_zip:52-86`), mais jamais évalué. Deux comportements non signalés : `move_lab_runs` saute **silencieusement** tous les sous-répertoires (l. 101-102) et tous les fichiers dont la cible existe déjà (l. 104-105) → perte de données silencieuse.
- **`api/tasks/setup_reference.py`** (408 l.) — sélection du truth-set GIAB via `GIAB_SAMPLES` + `extract_base_sample` (match `startswith` lâche, l. 68-70). Un mauvais mapping benchmarke contre **le mauvais truth set** → recall/precision/F1 corrompus, **mode d'échec sur la sortie scientifique principale**.
- **`security.py`** comme module — confirme : `hash_password` sans consommateur (aucun login), users/mots de passe décoratifs ; défaut ADMIN.
- **3e mécanisme de bootstrap** : l'audit n'a nommé que 2 des 3 (`init_db.py` + Alembic), manquant `create_db_tables.py` comme pont.
- **Tests** : couverture affirmée mais inexistante — seuls `test_transfer_*.py` existent ; **zéro test** sur pipeline/uploads/runs/dash.
- **`callbacks.py`** seulement sondé (finding l. 450-456) ; le graphe complet de callbacks + `home.py`/`index.py`/`visualization.py` non examinés (c'est là que se cachait le crash Summary corrigé).
